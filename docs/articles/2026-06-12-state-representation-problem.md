# 世界模型有一个状态表示问题

VAST 近期宣布拿了近两亿美元做 "Project Eden"，定位是 AI 3D world generator。OpenAI 的 Sora 不停往"世界模拟器"的方向 reposition，DeepMind 的 Genie 喊出 "foundation world model"。这些公司都在说"世界模型"，但他们指的不是同一个东西 —— 实际上，他们指的是三件完全不同的事。

主流叙述把这三件事压成同一句话，因为"世界模型"这个词太好用 —— 模糊到任何在做生成式 3D 的人都能往自己身上套。但这三件事之间的区别，决定了未来十年这个市场的版图。我想把它们拆开。

## 三个 rung

世界模型有三种实现方式，按 *世界状态如何演化* 区分：

**rung 1 — 脚本化 (scripted)**：世界状态是一段时间轴，`f(t) → state`。给定时刻 t，世界长成那个样子。p5 sketch、Pixar 动画、过场动画、shader 时间动画都是这一种。没有未来，只有播放。

**rung 2 — 模拟化 (simulated)**：世界有一个状态 + 一个 transition function，`f(state_t, action) → state_t+1`。每一刻的世界由上一刻的世界 + 这一刻发生的事计算出来。游戏引擎、物理模拟、CFD 都是这一种。未来是被算出来的。

**rung 3 — 学习化 (learned)**：transition function 本身是一个训练出来的模型。Genie 是这条路线最纯粹的例子。未来是被神经网络预测的。

这三个 rung 不是优劣排序，是不同的工程范式。但它们对"世界"这个词的承诺差别非常大：rung 1 给你一段播放，rung 2 给你一个可交互空间，rung 3 给你一个统计的可能性云。

主流玩家分别在哪里？

- **Sora / Veo / Runway** —— 视频生成，往 rung 3 走，但目前还是 rung 1 的高级版（生成一段播放，不能交互）
- **Genie** —— 真正在赌 rung 3，"agent 在生成出来的世界里做动作，模型预测下一帧像素"
- **逆矩阵科技 (Physis)** —— 国内最 ambitious 的 rung 3 押注。2026-06 超亿美元融资（BAI + 蚂蚁 + 经纬 + 五源 + 高瓴），定位"通用世界基座模型"，主张 AI 第三次范式跃迁是"预测下一个物理状态"。他们的差异化在于显式拒绝像素空间 scale up，改在**"物理隐空间"**做 scaling（编码力 / 速度 / 摩擦等抽象表征，剥离视觉冗余），并用 RLVR + 物理验证沙盒外挂 validator。**这是 sampled family 内部的一次抽象层上移** —— 从像素采样升到 latent vector 采样，比 Sora 更接近符号，但仍然在 sampled 这一边（法则在训练时被绑定到模型权重，不是 runtime 可编辑的代码）
- **LiberAI** —— 与 Physis 同日（2026-06-17）宣布数亿元 Pre-A（顺为领投 + 凯辉 / 红杉中国 / 真格 / 元禾），同样定位"物理世界基础模型"。创始人刘松铭已 ship 过两代具身基座（RDT-1B、RDT-2，均比硅谷竞品早一个月），路线是**"原生物理模态预训练"+ 模型 / 数据 / 硬件全栈闭环**。值得注意的是他们在融资稿里 explicit 把 world model 路线分成了 5 派 —— 视频生成（Sora / Genie）、3D 空间智能（World Labs）、物理引擎（NVIDIA Isaac / Genesis，"把规律和数据写进代码"）、隐空间预测 JEPA（LeCun）、世界动作模型 WAM（DreamZero）—— 自己定位为第 6 条"原生物理模态预训练"。**这份 5-派 self-classification 本身值得记下来**：industry 自己已经承认"物理引擎派"是 valid 路线之一，只是这 5 派全部落在 sampled / learned 这一边，没有任何一家公开 framing 是"符号家族 + 运行时绑定 + LLM 执行者"。Physis + LiberAI 同日双下注、投资人 cluster 完全不重叠，说明这不是单点事件，是国内一线 VC 的行业级 race
- **VAST Eden** —— 最值得细看的一家。他们公开的是三层解耦架构：结构化状态层、条件接口层、生成式渲染层 —— 也就是说，他们明确选择了**显式状态**这一边，这个判断值得尊重。但 transition 那一半，官方措辞是"下一步要搭建专属状态转移模型" —— 将来时。所以今天的 Eden 准确说是：**rung 2 的状态底座已交付，转移函数还在融资叙事里**。两亿美元买的不是一个完成的世界模型，是一个完成的 State 架构 + 一张通往 Transition 的路线图
- **Unity / Unreal + AI 工具链** —— rung 2，但人类工程师写 `Update()`，AI 只参与 asset 生成
- **Tripo / Hunyuan3D / Trellis / CSM** —— 都是 rung 1 的 asset 生成。这条赛道（asset generation as a service）是真生意、有真现金流 —— 但它解的是"AI 替代美术工时"，不是世界模型，虽然 marketing 经常往这边靠

最近 1-2 年最显眼的 framing 转向是 rung 3 —— Sora、Genie 把"我们在学世界"作为旗帜。这是 OpenAI / DeepMind 这种 scale 决定论玩家最自然的押注：给我足够多视频，我学出一个世界。

但这条路线里有个隐藏的问题没人在公开讨论。

## 隐藏的分叉：状态怎么表示

rung 2 和 rung 3 都有一个 state，但 state 长什么样 —— **这才是真正的架构分叉**。

两个家族：

**符号家族 (symbolic)**：state 是一棵可代数复合、可被推理的表达式树。SDF（signed distance function）是其中之一；OpenSCAD、CadQuery、JSCAD、L-system、CSG tree 都属于这家。共同特征：world 是一段可被人类（或 LLM）阅读、编辑、推理的代码。布尔与域操作在这个家族里是封闭的 —— 树和房子做并集，结果还是一棵干净的表达式树，可以继续参与下一次运算。

**采样家族 (sampled)**：state 是某个 manifold 在某个分辨率下被采样的结果。mesh（顶点 + 三角面）、NeRF（神经辐射场）、Gaussian Splat、voxel grid、point cloud、video frame —— 都属于这家。共同特征：你拿到的是数据数组，不是公式。mesh 和 mesh 做布尔，得到的是常常非流形、需要 retopology 修复的残局 —— "CSG cleanup" 之所以是一门独立的工程学科，正因为这个家族在组合运算下不封闭。

这个分叉决定了一件事：**LLM 能不能 native 写它**。

LLM 的训练分布里有几百万行符号代码、几千万行函数复合、boolean 运算、L-system 描述。它 *没有* 被训练去手写 50000 个顶点的 vertex array。所以：

- LLM × SDF / OpenSCAD = native writing。"画一棵树" → 输出几行表达式
- LLM × mesh asset = RPC into a generator。"画一棵树" → 调用 `tripo.generate("tree")` → 得到一个 mesh blob，里面几万个浮点数 LLM 自己读不懂也改不了

VAST、Tripo、Hunyuan3D、Trellis、CSM 这些公司的输出底物全部是 mesh。无论他们模型内部塞了多少 diffusion、transformer、3D VAE、multi-view consistency loss —— **模型内部的数学很丰富，但数学没贯穿到输出层**。你拿到 mesh 那一刻，数学性就被冻结成数据了。

这就像 JPEG 算法本身是数学，但 JPEG 文件不是 —— 它是数学的某个执行结果。LLM 能写算法（写一个 JPEG 编码器），但 LLM 不能"编辑"一个 JPEG（除非借工具）。这是同一种结构差别。

这一点在视频模型上更极端：Sora 输出像素帧，**没有可读取、可编辑的显式 state** —— 状态隐含在权重和生成上下文里，每一次生成是从分布里采样一段 trajectory，采样结束即蒸发。下一次生成是另一段独立 trajectory，没有从前一段继承的可编辑中间产物。（rung 3 的前沿在改善生成内的一致性 —— Genie 已能维持分钟级的场景连贯 —— 但"模型内部隐式记得"和"用户手里有一份可 diff 的文档"是两种承诺。）

## 状态住在哪：一份文档，还是五块互不相识的内存

表示形式是状态问题的上半场。下半场是一个几乎没人问的问题：**运行时，世界状态究竟住在哪几块内存里？**

拿今天组装一个可交互 3D 世界的标准栈来回答 —— mesh 渲染器（three.js / Unity）+ 外挂物理引擎（Bullet / PhysX）。这是 rung 2 经引擎实现的通行路线，也是 asset 生成公司的输出最终要落进去的地方。在这个栈里，每一帧的 "world state" 实际由这些 buffer 共同构成：

1. **GPU 显存里的 mesh data**：每个 mesh 的顶点 / 法线 / UV 数组，住在 VBO/EBO 里。改一个顶点要走 GPU 同步；序列化要从显存 read-back，慢且阻塞渲染。
2. **CPU 堆上的 scene graph**：每个节点的 position + quaternion + scale，父子矩阵嵌套，改一个 transform 要 mark dirty 等下一个 render pass 重算。
3. **物理引擎的私有 C++ 堆**：刚体的速度 / 惯性张量 / 休眠状态，约束求解器逐接触点累积的冲量，broadphase 的动态 AABB 树，跨帧缓存的接触流形（warm starting 的来源）。这一块对宿主程序**不可见** —— 它是求解器的私有记忆。
4. **动画系统**（如有）：clip 游标、blend 权重、骨骼相对 rest pose 的 delta 矩阵。
5. **粒子系统**（如有）：位置 / 速度 / 寿命数组，常驻 GPU 由 compute shader 驱动。

这五块东西分散在 GPU 显存、JS/C# 堆、物理引擎 C++ 堆、动画系统里，**互相不知道彼此的存在**。没有一个统一的 "World" 对象 —— "世界状态"是一个你必须用胶水代码在脑中拼合的虚构概念。

这不是美学洁癖，它级联到产品层。想"保存这个世界"？你要从 GPU read-back 顶点、序列化整棵 scene graph、调物理引擎的专有 API 把求解器状态编码成二进制 blob、再抄下动画游标和粒子 buffer —— 得到的不是一份文档，**是一个 zip 包**：mesh 文件 + scene JSON + 物理二进制 blob + 动画状态的捆绑。这个 zip 包不可读（物理 blob 是二进制）、不可 diff（两个 blob 没法人类对比）、不可 fork（"在第 300 帧分叉"要 deserialize → 重放 → re-serialize → 复制，每步都贵且脆）、不可一条链接分享、不可作为多人世界的同步基底。于是时间旅行调试、分支时间线、世界作为可交易 IP、演化历史作为出处证明 —— 每一项都变成独立的工程项目。

而在符号 + 纯函数路线里，这些是同一个数据结构的免费副产物：状态是一份 JSON 文档，演化日志是确定性的 action 序列，我们实测一个 600 tick 的粒子世界存档是 15.5 KB 的纯文本。回到第 300 帧是一次数组切片，分叉是截断加新分支，分享是发一个文件。**"世界即文档"不是比喻，是字面意思。**

（公平地说：VAST Eden 的结构化状态层正是对这种碎片化的一次反叛 —— 这也是前文说他们"在显式状态这边下了对的注"的原因。但只要动力学经由外挂求解器实现，第 3 块私有内存就仍然在那里，文档就仍然缺一角。）

状态住在哪，是第一根轴。第二根轴是法则 —— 它在什么时候被绑定。

## 编译时、训练时、运行时

把整件事压到一个轴上：看 *世界的法则是在什么时候被绑定的*。

**物理引擎在编译时绑定法则**。Bullet、PhysX、Rapier 的整个 solver 用 C++ 写在引擎源码里。重力的方向、摩擦的形式（Coulomb 锥）、约束的"必须为真"（Signorini 条件）—— 编译进库的时候就定死了。运行时你能动的只有参数：`setGravity(0, -12.7, 0)` 是合法的，但"让摩擦遵循另一种定律"不是。**引擎给你的是一族法则上的参数旋钮，不是法则的著作权。**

**学习模型在训练时绑定法则**。Sora、Genie 这类模型全部的物理（如果有的话）来自训练语料里出现过的物理。1g 重力的物体怎么运动，它学得到；月球上 0.16g 它也学得到（如果训练集里有月球视频）；但"假如光速是 100 m/s 会怎样" —— 不可能，因为没人拍过。学习模型只会算它见过的或被写进数据里的那一族世界。

这不是 Atlas 在外面攻击对手。逆矩阵科技在自己的官宣里就明确说："**纯生成式的路径只有生成能力、缺少验证能力，容易出现穿透、失重这类'物理幻觉'**。"一个刚拿超亿美元、最 ambitious 的中国 rung 3 玩家，**在自家融资稿里承认了这条路线的结构性 symptom**。他们的解法是用 RLVR 物理验证沙盒在外面包一层 validator —— 修补 symptom，没换 architecture。但"穿透 / 失重"是物理引擎里 Signorini 约束不被满足的具体表现；解决这类问题的本征方式是**法则可在运行时被声明、被验证** —— 也就是运行时绑定，而不是训练时硬塞 + 沙盒后修。

**规则运行时在运行时绑定法则**。Rule-writable 系统的 state 演化是用户 / LLM 在运行时编辑的代码。重力 ≠ 9.8，改一行；潮汐力是一条规则，加 5 行；时间膨胀是一个 `timeScale(p)` 函数，加 3 行。法则跟世界一起在文档里出现，跟世界一起被保存、被版本化、被分发。

这就是最压缩的世界模型 framework：

> **物理引擎在编译时绑定法则。学习模型在训练时绑定法则。规则运行时在运行时绑定法则。**
>
> **一个世界是一套规则；另一个世界，是一个 diff。**

第二句是第一句的工程含义。如果法则在运行时绑定，那么把地球切换到火星不是"重新生成一个世界"，是"打一个 patch"。`gravity 9.8 → 3.7`，几行代码。

## Outer Wilds：在 Unity 里手写了一个 mini 引擎

游戏圈知道这个故事，外面的人少。

*Outer Wilds*（Mobius Digital，2019）是一款基于 n-body 引力的太空探索游戏：行星互相绕转，玩家的物理由当前主导星球的引力场决定。**他们用 Unity 做的**。Unity 默认是 Newtonian 单一向下重力。所以他们做了什么？

他们绕开 Unity 的内置重力系统，**在 Unity 内部用 C# 手写了一套 n-body simulation**。从此 Unity 对他们来说不是物理引擎了 —— 只是渲染引擎 + 一个 update loop 入口。法则全部在他们手写的引力体系里。

这是"在 game engine 默认架构里争取异质物理"的活案例：**一支天才团队，要做引擎架构没预想的事，不得不在引擎里重新发明一个小型规则运行时**。Outer Wilds 团队是聪明人；他们做这个不是为了"优雅"，是因为 Unity 的默认假设逼着他们做。

如果你把他们当时做的事抽象出来 —— state 显式、规则可写、每帧 fold over rules —— 那基本就是 rung 2 范式的骨架。他们没有 LLM，他们手写 C#。

顺着这个案例，必须诚实地点名这条路线今天最近的邻居：**LLM × 现代 ECS 引擎**。Bevy 这类引擎里，system 本来就是作用于数据查询的小函数，有严格的 loop contract 和调度纪律，而 Rust 是 LLM 写得动的语言 —— 用 Claude Code + Bevy，今天就能复刻 Outer Wilds 团队当年手工做的事。这是 rung 2 LLM-native 命题最强的现实竞争形态，比任何 mesh 生成公司都近。它和符号文档路线的真实分界在四处：ECS 的 World 是内存中的 archetype 表，序列化是出了名的痛点，而符号路线的**状态本身就是一份 JSON 文档**；确定性在 Rust + 浮点的跨平台现实里仍要自己挣，而在纯函数 tick 里是默认；mesh/3D-locked vs 维度无关；原生应用 vs 浏览器零安装。换句话说：调度纪律他们更成熟（值得学），世界即文档他们给不出（这是分界线）。

## 一颗潮汐锁定的海洋行星 —— 但杀招不是浪

到这里我们手上有了一个 framework + 一个 evidence。接下来用一个 thought experiment 测试它的极限。

我选物理学家 Kip Thorne 在 *The Science of Interstellar*（2014）里详细推导过的那颗"潮汐锁定海洋行星" —— 大众也许更熟悉它在那部电影里的名字。它有三个区分性物理特征：

1. **1.3× 地球表面重力**
2. **来自附近黑洞的潮汐力主导海洋动力学** —— 那一公里高的浪不是随机的，是 ~1 小时一次的潮汐周期
3. **重力时间膨胀：表面 1 小时 = 外部 7 年**

电影 IP 我不引，但物理本身是 Thorne 公开发表的学术，可以自由讨论。

现在问三种架构：如果一个用户想"访问"这颗星，每种架构要做什么？

| 特征 | 物理引擎 | 视频模型 | 规则运行时 |
| --- | --- | --- | --- |
| 1.3g 重力 | `setGravity(0, -12.7, 0)` —— 一行 | 没有 g 这个旋钮：无法指定，无法验证 | `gravity: 9.8 → 12.7` —— 一行 patch |
| 潮汐力 | 绕过引擎管线手写 custom force —— 工程量但能做 | 同上，无法把"周期一小时的潮汐"作为可验证约束写进去 | `forces` phase 一条 ~5 行规则 |
| **时间膨胀** | **要动引擎的地基假设** | **结构性 OOD** | **`timeScale(p)` 规则，几行** |

前两个特征引擎咬咬牙都能扛。视频模型在前两行输掉的方式值得注意 —— 不是"画不出来"（给它足够的科幻片它能画出*看起来*更重的坠落），而是**没有任何接口让你绑定一个值、也没有任何方式验证输出服从了它**。法则在训练时已经绑定完毕，用户手里没有旋钮。

**第三行才是照妖镜。**

时间膨胀的物理含义是：**dt 本身成为空间的函数**。靠近黑洞的区域以 1/61320 的速率演化，远离的区域以"正常"速率演化。你站在浪的边上、伙伴飞船在轨道上等着，你这边过 3 小时，他们那边过 21 年。

这件事对物理引擎意味着什么？**整个引擎架构建立在"全场共享一个 dt"之上**。每一行 solver 都假设这个 dt，每一个 broadphase cache 都用这个 dt 更新，每一对接触约束都用这个 dt 求冲量。要支持空间变化的 dt，等于动引擎的地基。Bullet、PhysX、Rapier 全部如此。

对视频模型呢？更绝望。这类模型学的是"看起来像物理"，但**它从来没见过差速时间流作为可交互系统**。给它一万小时电影素材它学到的是"巨浪场景的电影质感"，*不是*一个时间流自洽、潮汐周期自洽、玩家进去可以触发反事实的世界。反事实物理对统计模型是结构性的分布外 —— 它只能预测训练分布里出现过的下一帧。

而在规则运行时里？时间膨胀是什么？给每个 subject 一个 `timeScale` 参数，让所有 forces / integrate 规则用 `dt × timeScale(p)` 步进。几行代码。**因为状态是显式的，每个区域的本地时钟可读可审计可序列化**。

我不夸大：多速率区域之间的物理交互（快区物体撞进慢区）确实有真正的设计难题要解 —— 守恒律怎么走？两个时钟怎么对齐？但**它在这个范式里至少是可表达的**。在另外两个范式里，它连出生的语法都没有。

把三种回答排成一行：

> **从地球到这样一颗星的距离 —— 在物理引擎里是一次 solver 改造工程，在视频模型里是一次不可能的 OOD 泛化，在规则运行时里是 ~15 行 patch（`gravity 9.8 → 12.7` + 潮汐力规则 + `timeScale(p)` 规则）。而且这个 patch 本身就存在 savegame 里，任何人能 diff 出"这个世界和地球差在哪三条法则"。**

这就是 binding-time 这个轴的杀伤力。

## 隐藏的 TAM：虚构世界的物理原型

退后一步看：rung-2 规则运行时这个范式服务谁？

答案出乎意料地大 —— 但目前没人在做产品。

**Kip Thorne 给 Christopher Nolan 做的事 —— "设定一套不同的法则，然后推演后果"** —— 是科幻作者、游戏世界观设计师、影视前期设计师、TRPG 系统设计师、硬科幻物理插画师每天的核心工作。他们目前用的工具：纸、笔、白板、Mathematica、自己的直觉。

没有任何工具让他们"写下法则、按下播放、看到涌现"。Sora 不行（学不到没拍过的物理），Unity 不行（架构假设固定，除非像 Outer Wilds 一样在里面重新发明运行时），Tripo / Hunyuan 更不行（只生成 asset）。

这个市场的上游是**所有人爱的 game world** 和**所有可信的硬科幻物理设定**。它现在被一群高学历 + 高直觉 + 高时间成本的专家手工服务。一个 rule-writable simulator 即便只覆盖这套 workflow 的 30%，影响也会 cascade 到下游一整圈创意产业。

把这个市场放进 LLM × symbolic representation 的整体 TAM 里：在 editorial illustration（杂志、海报、绘本）、PPT 视觉、emoji/icon 之后，**虚构世界 worldbuilding 是 LLM × symbolic 这条路最自然的下一站**。它的客户单价更高、复利效应更强、文化辐射半径更远。

## 收尾

绕回开头：VAST 拿了近两亿美元做 AI 3D world generator。Sora 朝"世界模拟器" reposition。Genie 喊 "foundation world model"。

到这里你应该能看出来，他们仨在解的不是同一个问题。VAST 在显式状态这一边下了对的注，但转移函数还在路上，他们今天的客户是游戏工作室；Sora 在赌 rung 3 像素学习，客户是内容消费者；Genie 在赌 rung 3 + agent action loop，客户暂时还是研究员。

**rung 2 LLM-native 这一块 —— 法则在运行时绑定、状态是符号文档、demo 是改三行代码看世界变化 —— 目前没有大玩家把它当主赛道**。离它最近的现实形态是 LLM × ECS 引擎，但那条路给不出"世界即文档"。这块空着不是因为不值得做，是因为它需要的技术栈跟 rung 1 / rung 3 玩家擅长的不重合：你要懂符号几何（SDF），要懂 LLM × code 那一面，要懂 game engine 设计但要有意识地拒绝 game engine 的默认架构。三种技能交叉的人不多。

我和团队在做的 Atlas（开源仓 sdf-js，github.com/shaun8149/sdf-js）是这块的一次尝试。这篇文章不是为了 pitch 这个项目 —— 我更在意 framing 本身能不能立住。如果 framing 是对的，未来会有 N 个这样的项目；如果是错的，提前发现也好。

最后回到一个比喻。

Sora 这条路是把世界录下来再让神经网络学。Atlas 这条路是把世界的法则写下来再让代码演算。两条路都是真的突破，都需要做。但你能学的世界，限于被拍摄过的世界。

**能被拍摄的世界才能被学习。能被书写的世界才能被模拟。而能被书写的，不止是世界的形状，还有世界的法则。**
